マルチタスクでは、同時に実行するといっても、厳密には本当に同時に実行するわけではありません。 タイムスライスといって、「プログラムAをちょっとだけ実行して中断」「プログラムBをちょっとだけ実行して中断」「プログラムAをちょっとだけ実行して中断」「プログラムBをちょっとだけ実行して中断」…という繰り返しで、全体的に見るとプログラムAとプログラムBが同時に実行しているように見せかけているのです。 そのため、「同時(分秒)まったく同じタイミング」と言っても、ナノ秒単位でみれば、実行の前後関係が発生します。 (マルチCPUの場合は、それぞれのCPUが別々にプログラムを動かせるため、本当の意味で同時に複数の処理が行われることがありますが、OSの機能によっては、問題が発生しないように「複数のCPUで並列処理」せずに「一つのCPUが順次処理」するようになっていますので、今回の話では考える必要はありません) 以上を踏まえた上で、シンボリックリンク作成を同時に複数実行した場合の挙動についてですが、 シンボリックリンクを作るOSの機能である symlink システムコールは「アトミック」なものになっています。 アトミック(原子的)とは、「分子は原子に分割できるが、原子はそれ以上分割できない」ように「これ以上の細かい機能に分割することのできない」ものであり、上述の「ちょっとだけ実行しては中断する」というタイムスライス処理によって「処理途中で中断する」ことがありません。 実行をし始めたら中断されることなく最後まで処理を進めるので、その結果は「完全に成功するか、完全に失敗するかの二択」であり、中途半端な状態が発生しません。 そのため、「秒単位でまったく同じタイミングでシンボリックリンク作成を同時に実行」しても、ナノ秒単位でも見ると「symlinkシステムコールの呼び出しには前後関係があり」、「先にsymlinkシステムコールを呼び出した方が成功する」ということになります。

一般的に、OSのインストールディスクのISOイメージファイルを USBメモリーにインストールするという行為は行ないません。 ISOイメージファイルを、DVD-Rに書き込む作業は ISOイメージファイルの種類を問わず、インストールとは言いません。 であれば、ISOイメージファイルをUSBメモリーに書き込んで USBメモリーでインストールディスクを作成することは、インストールではありません。 そのインストールディスクで、別のUSBメモリーにインストールすることはできますが それは、別のUSBメモリーにOSをインストールしているのであって OSのインストールディスクのISOイメージファイルをインストールしているのではありません。 後者は、USBメモリーを、HDDとほぼ同等のストレージ装置として管理できる Linux系OSの特性を活かしたインストール方法です。 前者は、USBメモリーを光学ドライブ互換のストレージ装置として インストールもしくはライブDVD相当としての一次的な利用を目的とします。 質問内容から、目的が特定できないわけですが これは、目的を達成するために、何をすべきかを問わず 「こうすれば目的を達成できるはずだけど、その手順がわからない」という 勘違いの視点から質問しているためです。 こういったサイトでは、そういった質問は非常に多く 混乱と、誤った解答を、おびただしいほど生じさせています。 つまり、最終的にArch Linuxを USBメモリーで、不特定のPCで利用することを目的としているのか? それとも、内蔵HDDにインストールして 一台のPCをArch Linux用にしようとしているのか? その点を主題とした質問をすべきでした。 もっとも、Arch Linuxの性質から、Arch LinuxをUSBメモリーにインストールして 不特定な仕様のPCで利用することは向いていないはずです。 不特定な、同じ仕様のPC群であれば、なんとかなるような気はします。 これは、Archが自動認識任せではなく、最小構成を企図した 手動設定を前提とした運用を志向しているように見えるからです。

apt-get というコマンドは、 Debian 式のパッケージ管理ソフト apt の１コマンドです。 最近は、apt という名前でも同じことをやる様になったみたいです。 それは、おいといて、 RedHat 系は、yum というまた別のパッケージ管理ソフトを使っています。 でも、原理はだいたい同じです。 Debian 系 apt の場合、 /etc/apt の中に sources.list sources.list.d/xxxx.list とかいう、リポジトリと呼ばれるファイルが入っています。 中身を見てみると、 deb http://mirrordirector.raspbian.org/raspbian/ jessie main contrib non-free rpi こんな事が書いてあります。 まず、 apt-get を実行すると、 http://mirrordirector.raspbian.org/raspbian/ というサーバーの中から、jessie main contrib non-free rpi に相当する設定ファイルが無いか捜して拾ってきます。 その取り出したものには、 何処の国に実体が置いてあるサーバーがあるか？ が書いてあるのです。 そこで、アクセスしたアドレスから国を割り出して、一番近いサーバーを見つけて、そこからダウンロードするのです。 これは、RedHat 系の yumでも、ファイルの名前が違うだけで、ほぼ同じことをしています。 ですから、わざわざユーザーが自分で良さそうなサーバーを捜さないでもよくなっているのです。 簡単ですが、結構分かりにくいですね。 ちなみに、これ以外にそのサーバーが本物か？ダウンロードされたものが正しいか？の検証も自動でやってくれます。 結構便利ですね。

典型的なインストール時のメッセージを読まないで 自分で特殊で異常で役に立たない構成を作ったというだけの状況です。 そのUSBメモリーを、別のPCに挿して起動しようとしても やはり起動できないという状況になっているはずです。 これは、ブートマネージャーが何であるのかを 理解していない初心者が、メッセージの意味を軽視して 行なうことでも知られている状況です。 問題はシンプルなもので、基本的には、Windowsでブートマネージャーの修復をすれば WindowsをクリーンインストールしたシステムのMBRは 元通りに戻せるはずです。 ですから、この解決方法は、Windowsのカテゴリーの問題で 実践的な対処方法は、第三者には想像することが難しい問題です。 問題が難しくなるのは、Windowsユーザーの多くが Windowsのクリーンインストール体験が無く またそれができるインストールディスクを所有していないことも多いということです。 MBRにPCメーカーが特殊なブートマネージャーを導入している機種もありますから そうなると、Windowsのブートマネージャー修復で回復することができません。 自分で上書きして消滅させた旧来のブートマネージャーを どこかから入手できなければ、元に戻せないということにもなります。 メッセージの意味を知らないまま 不用意に行動した自分が、それを消したことを認識して 今後は、慎重な行動をお薦めします。 内蔵HDDのパーティション分割を施して Linux系OSをHDDにインストールし ブートマネージャーもLinux系OS用のものを改めてMBRに入れれば 通常のデュアルブート構成になるので USBメモリーへの依存は解消されます。 ただし、デュアルブートは非常に特殊な運用ですから 新しい問題を生み出すこともありえますし 将来的なトラブル要因を作ることになる恐れもあります。 ついでにどういうことがおきているのかも説明しておきますが 簡単に言えば、高機能化されたブートマネージャーは 446bytesしか無いMBRに収まらず 他の領域への部分的な分割収容が行われることがあるので その領域をUSBメモリーとして切り離せば、当然起動しないというものです。 Windowsのブートマネージャーも、ブート設定をC:ドライブ等に保存しています。 (Windows8世代ならシステム領域として、さらに別になっていたりします) こういったMBRからはみ出してしまうことは、よくある仕様で デュアルブートのような特殊な運用を計画し 充分な知識や注意が無いまま作業を行なった結果、問題を起こす程度です。 Windowsの場合はブートマネージャー自体がWindows以外を想定していないので WindowsとWindowsのデュアルブートをしないかぎり体験することは無いかもしれません。 Linux系OSの場合は、一般的にインストール先となるストレージ装置に ブートマネージャーの設定ファイル等を収容しています。 ですから、それに該当するUSBメモリーが無いと 内蔵HDDのMBRに書き込んだブートマネージャーは 設定ファイルが無いため、途中で動作不能となります。 内蔵HDDにLinxu系OSを導入すれば、そのミスマッチは解消されますし より高い知識があれば、起動設定だけをHDDに移植する手法もあります。 ただし、それはブートマネージャーの仕様や仕組みを理解し 手作業で設定ファイルを書き換える必要があり 将来的なトラブル要因もあるので 個人的には、そんなバカみたいな設定はやるだけ無駄だと思います。

Linux ではドライバが必要ないというわけではなく、単に Ubuntu が新しいデバイス向けのドライバを多く同梱しているというだけです。 まず、Windows と Linux ではドライバの配布形態に違いがあります。 Windows ではデジタル署名したバイナリのドライバが配布されますが、Linux では数多くのディストリビューションが存在する上、バージョンアップの頻度も高いためにバイナリのドライバを配布することは非常に難しいです。 そこで、Linux 用のドライバはソースコードとして配布されることになりますが、もし Linux の該当サブシステムのメンテナがよしと言えば簡単に Linux のメインツリーに取り込んでもらうことができます。 もちろん Windows でも NIC 等の重要なドライバは同梱します。ですが、Windows と（一般的な）Linux ディストリビューションではリリース周期が違います。 Windows は数年おきですが、Ubuntu の場合は半年に一度ですね。 上流の Linux カーネルのメインラインに至ってはおよそ 10 週間に一度新しいバージョンがリリースされます。 これらの理由により、Ubuntu では追加でドライバをインストールすることなく多くのデバイスが使えるようになっています。
